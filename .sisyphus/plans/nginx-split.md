# nginx-split: Decouple nginx from Go and Split into Separate Container

## Context

### Original Request
Implement the 5 tasks from `nginx-split.md`:
1. Serve Switchboard from Go directly
2. Split nginx into its own container
3. Update compose.yml to reflect changes
4. Rename NGINX_ENABLED → NGINX_CONF_GEN_ENABLED
5. Update documentation

### Interview Summary
**Key Discussions**:
- **nginx reload strategy**: Shared volume + `docker exec` via Docker API. Go writes config to a shared volume, then uses the Docker API (which it already imports) to exec `nginx -s reload` in the nginx container.
- **UI serving from Go**: Enhanced Go file server with SPA fallback middleware and cache headers for `/_next/` assets. Keep serving from disk at `/app/ui`. No embedding.
- **WebSocket handling**: Go already handles WebSocket upgrades directly via gorilla/websocket. No additional middleware needed since nginx will no longer sit in front for Switchboard UI traffic.

**Research Findings**:
- Go already has `mux.Handle("/", http.FileServer(http.Dir("ui")))` in `handlers.go:74` — baseline file serving exists.
- nginx currently serves UI on port 80 as `default_server`, proxying `/api/`, `/healthz`, `/metrics`, `/ws` to Go at `127.0.0.1:8069`.
- The `nginx:1.27-alpine` base image contains both nginx and the Go binary — this is the monolith to split.
- `deploy/entrypoint.sh` starts both processes and monitors them.
- The Docker Go client (`github.com/docker/docker`) is already a dependency and supports `ContainerExecCreate`/`ContainerExecStart` for remote exec.
- The UI is a Next.js static export (`output: 'export'`) — pure static HTML/JS/CSS files in `/app/ui`.

### Metis Review
**Identified Gaps** (addressed in plan):
- **nginx container name discovery**: Go needs to know which container to exec into. Use `NGINX_CONTAINER_NAME` env var (default: `switchboard-nginx`).
- **Graceful degradation**: If nginx container is not running, config generation should still work (write to volume) but skip the reload exec. Already handled by error logging in current code.
- **Port mapping shift**: Go will now serve on port 80 directly (was 8069). nginx container needs its own port mapping for reverse proxy traffic.
- **Config generation path change**: `NGINX_GENERATED_CONF` default path should work with the shared volume mount.
- **Backward compatibility of env var rename**: Consider supporting both old and new env var names during transition. Plan includes deprecation log.

## Work Objectives

### Core Objective
Decouple the Switchboard Go backend from nginx so they run in separate containers. Go serves the UI and API directly. nginx runs independently for reverse-proxy duties with config generated by Go via a shared volume.

### Concrete Deliverables
- Modified Go backend that serves UI with SPA fallback and cache headers on port 80
- Separate nginx container in `compose.yml`
- Shared volume for nginx config generation
- Docker API-based nginx reload from Go
- Renamed `NGINX_ENABLED` → `NGINX_CONF_GEN_ENABLED` across entire codebase
- Updated Dockerfile (Go-only, no nginx)
- New `Dockerfile.nginx` for standalone nginx container
- Updated entrypoint script (Go-only)
- Updated documentation (README.md, examples/README.md, PERFORMANCE.md, backend/README.md)

### Definition of Done
- [ ] `docker compose up --build` starts both containers successfully
- [ ] Switchboard UI loads at `http://localhost` served by Go
- [ ] API endpoints (`/api/containers`, `/api/config`, `/healthz`, `/ws`) work via Go
- [ ] WebSocket connections work directly to Go
- [ ] nginx container runs separately and serves reverse proxy vhosts
- [ ] Go generates nginx config and triggers reload in nginx container
- [ ] All existing tests pass (`cd backend && go test ./...`)
- [ ] No references to `NGINX_ENABLED` remain (only `NGINX_CONF_GEN_ENABLED`)

### Must Have
- SPA fallback for the UI (serve `index.html` for non-file routes)
- Cache headers for `/_next/` static assets
- Shared volume between Go and nginx for generated config
- Docker exec-based nginx reload from Go
- Backward-compatible deprecation message for old `NGINX_ENABLED` env var

### Must NOT Have (Guardrails)
- Do NOT embed UI files into the Go binary (keep filesystem serving)
- Do NOT change any frontend/UI code
- Do NOT change the nginx config generation logic itself (only HOW it writes and reloads)
- Do NOT change the API endpoints or WebSocket protocol
- Do NOT modify CI/CD workflows (.github/workflows/)
- Do NOT add new Go dependencies unless absolutely necessary (Docker API exec is already in the existing dependency)
- Do NOT remove the `connection_upgrade` map directive from nginx config (still needed for generated vhosts)
